## Goal and intent

- Goal: Expose a safe, workspace-scoped, role-gated Shopify integration management surface — list/detail/reauthorize/disconnect/manual-sync/scope-status routes, backed by new query and command modules, and `domain/shopify/results.py` + `domain/shopify/serializers.py` — so the React frontend can show and manage linked Shopify shops without ever seeing a secret.
- Business/user intent: Let an admin or manager see which Shopify shops are linked to their workspace, understand their health (scope/webhook status), fix problems (reauthorize, manual resync), and let an admin disconnect a shop cleanly — completing the intention plan's "Frontend integration intention" and "Router intention" sections using the backend built in phases 1-5.
- Non-goals:
  - Shopify OAuth callback changes (master phase 2, implemented) — this plan reuses `create_shopify_install_url` unmodified for reauthorization; see "Resolved decisions" item 6.
  - Inbound Shopify webhook route or HMAC verification changes (master phase 4, implemented).
  - Worker runtime internals or queue map changes (master phase 5, implemented and archived) — this plan only calls `create_instant_task` with phase-5's existing task types, exactly as any other command in this codebase already does for user-triggered background work (e.g. `sync_email_connection`).
  - New webhook subscription GraphQL logic — this plan only calls phase 3's existing commands (synchronously) or enqueues phase 5's existing task types (asynchronously); it authors no new GraphQL query/mutation.
  - Product/order business processing.
  - Historical imports.
  - Frontend UI implementation (React) — this plan is backend-only.
  - Production process-manager wiring.
  - Creation of remaining child implementation plans (phase 7 of the master plan).
  - Real Shopify OAuth app uninstall/revoke API calls — disconnect in this phase clears local state and stops future webhook subscriptions; it does not call any Shopify "revoke access" endpoint, since no such infra capability exists in phases 1-5 and this plan does not add one.

## Resolved decisions

These design questions are resolved for this child plan by direct inspection of the actual codebase (not generic contract examples). File:line references reflect the state read on `2026-07-09`.

1. **Role/permission policy — confirmed, matches the requested policy exactly, no clarification needed.** `routers/utils/roles.py` defines `ADMIN`, `MANAGER`, `WORKER`, `SELLER` (unchanged since phase 2). `routers/utils/jwt_dep.py`'s `require_roles(allowed_roles: list[str])` is the sole enforced gate (confirmed: the generic `Role`/`BackendGroupPermissions` system from `28_roles_permissions.md` exists in JWT claims/`ServiceContext` plumbing but has zero callers of `ctx.has_permission()`/`ctx.require_permission()` anywhere in `services/`, and `routers/middleware/backend_permission.py` explicitly defers to `require_roles(...)` for any request carrying a known flat role — the granular system is unused scaffolding, not a second gate to satisfy). Applied exactly as requested: `GET /shops`, `GET /shops/{id}`, `GET /scopes`, `POST /shops/{id}/reauthorize-url` -> `require_roles([ADMIN, MANAGER])`; `DELETE /shops/{id}`, `POST /shops/{id}/webhooks/sync`, `POST /webhooks/sync` -> `require_roles([ADMIN])`. `WORKER` and `SELLER` are rejected from every route in this phase.
2. **`results.py` + `serializers.py` split — canonical pattern, not the emails/tasks shortcut.** `domain/shopify/results.py` defines frozen dataclasses: `ShopifyShopIntegrationResult`, `ShopifyWebhookSubscriptionResult`, `ShopifyScopeStatusResult` (fields per the Scope section's suggested lists). `domain/shopify/serializers.py` defines `serialize_shopify_shop_integration(row: ShopifyShopIntegration) -> ShopifyShopIntegrationResult`, `serialize_shopify_webhook_subscription(row) -> ShopifyWebhookSubscriptionResult`, `serialize_shopify_scope_status(...) -> ShopifyScopeStatusResult`, each converting an ORM row (plus, where needed, small derived values — see items 8-9) into the typed result. Routers/queries call `asdict(serialize_...(...))` to produce the JSON-safe dict, matching `46_serialization.md`'s documented `asdict()` convention. This deliberately follows the canonical documented contract rather than the emails/tasks codebase shortcut (see Scope > Assumptions) — a legitimate, explicit choice for this phase, not an oversight.
3. **Query pagination/soft-delete convention — matches the actual, locally-overridden contract.** `07_queries_local.md` overrides the generic cursor-pagination doc with **offset pagination**: `limit`/`offset` query params (default `limit=50`, max `200`), fetch `limit + 1` rows, `has_more = len(rows) > limit`, response envelope `{"shops": [...], "shops_pagination": {"limit", "offset", "has_more"}}` — matching `services/queries/emails/list_email_connections.py` exactly. Soft-delete filtering uses `ShopifyShopIntegration.is_deleted.is_(False)` (the boolean column, confirmed present on this table since phase 1 — not `deleted_at.is_(None)`, which some other domains use instead; the two conventions coexist in this codebase depending on which column the target table actually has, and `ShopifyShopIntegration` has `is_deleted`).
4. **List query does not filter by status.** `list_shopify_shop_integrations` returns every non-deleted shop for the workspace regardless of `status` (including `disabled`/`uninstalled`/`error`) — the frontend needs to show disconnected/broken shops too, not just active ones; the `status` field itself (already in the result) is how the frontend distinguishes them. Sort: `created_at.desc()` (newest first), matching the emails list-query precedent.
5. **Disconnect behavior — fully resolved, including event type.** `disconnect_shopify_shop` (ADMIN-only): loads the `ShopifyShopIntegration` row workspace-scoped by `shop_integration_id` (raises `NotFound` if missing or `is_deleted`); captures `previous_status = integration.status.value` before mutation; sets `status = ShopifyIntegrationStatusEnum.DISABLED` (not `UNINSTALLED` — `UNINSTALLED` is reserved for when Shopify itself reports the `app/uninstalled` webhook topic, a Shopify-initiated event; this route is a ManagerBeyo-initiated deliberate disconnect, so `DISABLED` is the semantically correct status, consistent with the active-like/inactive split every prior phase has used); sets `uninstalled_at = now()` (the only "disconnected at" timestamp column phase 1's schema provides — reused here regardless of the `DISABLED` vs `UNINSTALLED` status choice, since there is no separate `disabled_at` column); sets `access_token_encrypted = None` (safe — the column is nullable, `Text | None`, confirmed in phase 1's model, no `NOT NULL` constraint to violate); does **not** hard-delete the row (`is_deleted` stays `False` — a disconnected-but-still-existing shop must remain visible in `list_shopify_shop_integrations` per item 4, distinct from a genuinely deleted row); records exactly one `ShopifyIntegrationEvent(event_type=ShopifyIntegrationEventTypeEnum.DISCONNECT, severity=INFO, metadata_json={"action": "disconnect", "shop_domain": integration.shop_domain, "previous_status": previous_status, "new_status": "disabled", "remove_webhooks_task_id": task.client_id})`; then enqueues `create_instant_task(session=ctx.session, task_type=TaskType.SHOPIFY_REMOVE_WEBHOOKS_FOR_SHOP, payload=asdict(ShopifyRemoveWebhooksForShopPayload(shop_integration_id=integration.client_id)), event_client_id=event.client_id)` (async, not an inline synchronous GraphQL call — see item 7) rather than calling `remove_shopify_webhooks_for_shop` directly in the request path — the event is created first so its `client_id` can be passed as `event_client_id` on the task, and the task's own `client_id` is included in the event's `metadata_json` for two-way traceability (matching phase 5's own `event` -> `create_instant_task(event_client_id=...)` ordering in `_webhook_sync.py`). No secret or token value appears anywhere in `metadata_json`. **The exact `event_type` for this event was the one open question in this plan; it is now resolved per explicit instruction**: `ShopifyIntegrationEventTypeEnum` gains one new member, `DISCONNECT = "disconnect"`, added via an additive Postgres enum migration (`ALTER TYPE shopify_integration_event_type_enum ADD VALUE IF NOT EXISTS 'disconnect'`), matching phase 1's own migration idiom for this exact enum type (`app/migrations/versions/677ed7131bb2_create_shopify_integration_foundation.py`, confirmed as the type's origin: `SAEnum(ShopifyIntegrationEventTypeEnum, name="shopify_integration_event_type_enum", create_type=True)` in `models/tables/shopify/shopify_integration_event.py`). Rejected alternative: reusing `WEBHOOK_SYNC` with a `metadata_json` marker — rejected because a deliberate admin disconnect is not a webhook-sync event and would make the audit trail semantically misleading; `DISCONNECT` is a small, safe, additive, one-line migration and gives phase 7 (or any future admin audit view) a clean, greppable event type. See "Clarifications required" (now resolved) and Implementation plan step 2.
6. **Reauthorization reuses phase 2's `create_shopify_install_url` completely unmodified.** `create_shopify_reauthorize_url` (ADMIN/MANAGER): loads the `ShopifyShopIntegration` row workspace-scoped by `shop_integration_id` (raises `NotFound` if missing); extracts the row's already-normalized `shop_domain` (never accepts a client-supplied `shop_domain` for this route, closing the "arbitrary shop domain" concern by construction); constructs a fresh `ServiceContext(identity=ctx.identity, incoming_data={"shop_domain": integration.shop_domain, "redirect_after_success": "default"}, session=ctx.session)` and calls `create_shopify_install_url(fresh_ctx)` **as a plain Python function call** (not via `run_service`, since this command is itself already running inside `run_service`'s error boundary — nesting `run_service` calls would double-wrap `DomainError` handling for no benefit). No modification to phase 2's `create_shopify_install_url.py` is needed: it does not check for an existing active-like integration at OAuth-*state-creation* time (that check only happens at *callback* time, in `_linking.link_or_update_shopify_shop_record`'s `_active_conflict_stmt`), so calling it again for an already-linked shop in the same workspace works cleanly — the callback will `UPDATE` the existing row (via phase 2's `_same_workspace_shop_rows_stmt` lookup) rather than create a duplicate, and phase 2's own event-type logic already sets `event_type=REAUTHORIZE` (not `INSTALL`) whenever `existing.installed_at is not None`, so "reauthorize" is automatically labeled correctly with zero new logic. Returns the identical `{"install_url", "shop_domain", "expires_at"}` shape phase 2's `install-url` route already returns.
7. **No inline Shopify GraphQL calls from any admin route.** Both manual-sync routes and disconnect enqueue phase 5's execution tasks (`SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP`, `SHOPIFY_REMOVE_WEBHOOKS_FOR_SHOP`) rather than calling phase 3's `sync_shopify_webhook_subscriptions_for_shop`/`remove_shopify_webhooks_for_shop` synchronously in the request path. This keeps every admin route's response time independent of Shopify's API latency/availability, consistent with the explicit instruction that admin routes must not run GraphQL webhook sync inline, extended consistently to disconnect as well (not just the two sync routes) for the same reason. Each enqueue call mirrors phase 5's confirmed `create_instant_task(session=..., task_type=..., payload=asdict(...), event_client_id=event.client_id)` shape (see "Phase 5 implementation dependencies to verify before approval"): `enqueue_shopify_webhook_sync_for_shop`/`enqueue_shopify_webhook_sync_for_workspace` each record a `ShopifyIntegrationEvent(event_type=WEBHOOK_SYNC, metadata_json={"action": "manual_sync", ...})` first, then pass that event's `client_id` as `event_client_id` on the `SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP` task — the workspace-wide variant does this once per eligible shop, in one event+task pair per shop, not a single event for the whole batch.
8. **`scopes_status` is derived from the existing `status` enum, not recomputed.** Rather than re-running `compare_requested_and_granted_scopes` at serialization time (which would duplicate logic that already ran at OAuth-callback time in phase 2's `_linking.py`), `scopes_status` in the shop serializer is a simple derived string: `"outdated"` if `integration.status in {SCOPES_OUTDATED, NEEDS_REAUTH}`, else `"up_to_date"`. Single source of truth stays phase 2's OAuth-time scope comparison; this phase only reads the already-computed `status`.
9. **`webhooks_status` is a lightweight, read-only local summary — no GraphQL call.** Computed entirely from locally-stored `ShopifyWebhookSubscription` rows for the shop (no remote Shopify call): `"has_failures"` if any row has `status=FAILED`; else `"needs_sync"` if any enabled, scope-covered registry topic (per `SHOPIFY_WEBHOOK_REGISTRY`/`has_all_required_scopes`, phase 1) has no matching `ACTIVE` local row; else `"synced"`. This mirrors phase 3's own reconciliation logic read-only, for display purposes, and triggers no Shopify API call — a genuinely different concern from phase 3's `sync_shopify_webhook_subscriptions_for_shop`, which performs the same comparison but *acts* on it.
10. **`GET /scopes` supports both a single shop and a workspace-wide list via one optional query param.** `?shop_integration_id=<id>` returns a one-element list (or 404 if the id doesn't resolve in this workspace); omitting it returns scope status for every non-deleted shop in the workspace. ## Risks and mitigations

- Risk: This plan was originally drafted against an approved-but-unimplemented phase-5 design; phase 5's actual task-type/payload names or handler delegation shape could have differed.
  Mitigation: The dedicated "Phase 5 implementation dependencies to verify before approval" section has since been checked against the real, archived phase-5 code and its implemented summary (`2026-07-09`) — every assumed name matched exactly, with one additive signature detail (`event_client_id`) now folded into this plan's own design, mirroring every prior phase transition in this series.
- Risk: A future contributor "helpfully" adds `access_token_encrypted` (or another secret) to `ShopifyShopIntegrationResult` because it's a convenient field to have, silently reintroducing a secret leak.
  Mitigation: The result dataclass simply has no such field — there is no filter to bypass, only a field to not add; the serializer test in Implementation plan step 10 fails immediately if one ever is added.
- Risk: `disconnect_shopify_shop` calls `remove_shopify_webhooks_for_shop` synchronously in the request path "since it's right there," reintroducing the inline-GraphQL problem phases 3-5 were careful to avoid on the webhook intake side.
  Mitigation: "Resolved decisions" item 7 makes this an explicit, tested requirement (enqueue, not inline call) for disconnect specifically, not just the two sync routes.
- Risk: The disconnect event-type clarification is left unresolved and Codex invents a third option (e.g. silently reusing `ERROR`) without it being a documented decision.
  Mitigation: "Clarifications required" states both real options with a recommendation; this plan cannot move to `approved` until this is explicitly answered, same discipline as every prior open clarification in this series.
- Risk: `create_shopify_reauthorize_url`'s nested, non-`run_service`-wrapped call into `create_shopify_install_url` causes a `DomainError` raised inside the inner function to propagate differently than expected (e.g. double-wrapped or swallowed).
  Mitigation: Calling the command function directly (not via `run_service`) means any `DomainError` it raises propagates normally up through `create_shopify_reauthorize_url` to the *outer* `run_service` call the router made — exactly the same propagation phase 5's handlers already rely on for their own direct command calls (see phase 5's "Resolved decisions" item 6, the precedent this plan's item 6 explicitly mirrors).
- Risk: `GET /scopes`'s dual single-shop/workspace-wide behavior via one optional query param is confusing or under-tested compared to two separate explicit routes.
  Mitigation: Implementation plan step 10 explicitly requires both branches to be tested; the route list this plan targets (from the task) names only one `/scopes` endpoint, not two, so this is the minimal design that satisfies the stated route list without inventing an unrequested second endpoint.
