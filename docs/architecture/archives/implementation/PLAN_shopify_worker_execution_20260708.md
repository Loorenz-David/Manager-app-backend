## Goal and intent

- Goal: Implement the dedicated Shopify background execution layer — task types, payload dataclasses, a dedicated `queue:shopify`, a dedicated worker entrypoint, and handlers that delegate to the already-approved/implemented phase 1-4 commands — then connect the two event-only boundaries phases 2 and 4 deliberately left inert (`enqueue_shopify_webhook_sync_after_install`, `enqueue_or_record_shopify_webhook`) to real execution enqueue calls.
- Business/user intent: Turn the durable-but-inert signals from phases 2 and 4 (an integration event saying "webhook sync is pending," an intake row saying "a webhook was received") into actual background work — webhook subscriptions get synced after OAuth, and received webhooks get marked processed — using the backend's existing, already-battle-tested execution layer (`ExecutionTask`/`ExecutionPayload`, Postgres-authoritative state, Redis-transport queues, `task_router`, `worker_base.run_worker`), not a new parallel job system.
- Non-goals:
  - New Shopify OAuth behavior (master phase 2, implemented).
  - New webhook subscription GraphQL behavior beyond calling phase 3's existing commands (master phase 3, implemented).
  - New webhook HTTP intake behavior beyond connecting phase 4's existing boundary to enqueue (master phase 4, implemented).
  - Product/order business processing — this plan's `SHOPIFY_PROCESS_WEBHOOK` handler marks intake rows processed; it does not parse or act on order/product payload content (see "Resolved decisions" item 5).
  - Admin routes (master phase 6).
  - Frontend UI.
  - Historical product/order imports.
  - Production process-manager wiring (systemd/PM2/Procfile/docker-compose entries) beyond a local dev entrypoint and Makefile target — see "Resolved decisions" item 9; full deployment wiring is master phase 7's concern unless the master plan is found to require part of it here (it does not, per the phase 5 "Owns"/"Must not touch" split reviewed below).
  - Creation of remaining child implementation plans (phases 6-7 of the master plan).

## Scope

- In scope:
  - Shopify execution task types added to the existing `TaskType` enum (`domain/execution/enums.py`).
  - Shopify execution payload dataclasses in `domain/execution/payloads/shopify.py`, matching the existing per-task-type dataclass convention.
  - A dedicated Shopify queue, `queue:shopify`, mapped in the existing `QUEUE_MAP` (`services/infra/execution/task_router.py`) — no Shopify task type shares an existing non-Shopify queue.
  - A dedicated worker entrypoint, `workers/shopify_worker.py`, registering only Shopify task types (no mixing with unrelated workloads), following the exact `workers/notification_worker.py` shape already used in this codebase.
  - Handler registration for: processing one webhook intake row, syncing webhooks for one shop, removing webhooks for one shop, and reconciling one shop (an alias to the sync handler — see "Resolved decisions" item 2).
  - Connecting phase 2's `enqueue_shopify_webhook_sync_after_install` event-only boundary to a real `SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP` enqueue, additive to its existing event recording.
  - Connecting phase 4's `enqueue_or_record_shopify_webhook` event-only boundary to a real `SHOPIFY_PROCESS_WEBHOOK` enqueue, additive to its existing intake/event recording, and only for the `RECEIVED` outcome (never for `IGNORED`, never for a no-op/dedupe/unknown-shop/invalid-HMAC outcome).
  - Idempotency guards for every Shopify handler, following this codebase's established re-fetch-and-check-state pattern.
  - Retry/failure/backoff behavior via the existing, unmodified `worker_base.run_worker` machinery — no new retry system.
  - Safe structured logs (task id, task type, shop domain, outcome) — never access tokens or raw webhook payloads, continuing phases 2-4's logging discipline.
  - One Alembic migration adding the new `task_type_enum` values (`ALTER TYPE ... ADD VALUE`), following this codebase's exact existing migration idiom for every prior `TaskType` addition.
  - A local dev entrypoint (Makefile target) for the new worker, matching the existing `notification-worker`/`presence-worker`/`analytics-worker` targets.
  - Tests: task type/queue map registration, payload dataclass round-trip, handler registration, idempotency (duplicate task execution is a no-op), retry/failure classification, and both boundary-connection tests (sync-after-install enqueues exactly one task; webhook intake enqueues exactly one task only for `RECEIVED`, never for `IGNORED`/no-op outcomes).

## Resolved decisions

These design questions are resolved for this child plan by direct inspection of the actual execution-layer code in this repository (not the generic architecture contract examples, which use illustrative names like `my_app` that do not match this codebase 1:1). File:line references below reflect the state read on `2026-07-08`.

1. **Boundary 1 — post-OAuth webhook sync connects to real enqueue.** `services/commands/shopify/enqueue_shopify_webhook_sync_after_install.py` (phase 2) and the shared helper it and `handle_shopify_oauth_callback` both call, `services/commands/shopify/_webhook_sync.record_webhook_sync_pending`, are updated to additionally call `create_instant_task(session, TaskType.SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP, payload=asdict(ShopifySyncWebhooksForShopPayload(shop_integration_id=...)))` immediately after the existing `_events.create_shopify_integration_event(event_type=WEBHOOK_SYNC, ...)` call, inside the same transaction. The existing event recording is **not removed** — the event and the task both exist, matching how every other instant-task command in this codebase pairs a domain write with `create_instant_task` in one transaction (see `services/commands/location_tracker/enqueue_item_zone_push.py`). This resolves the "Master-plan deviation requested" note logged during phase 2's review, which explicitly deferred this exact wiring to phase 5 ("master plan phase 2 describes this as 'enqueue of webhook sync after successful link' — this plan implements the named boundary call site now and defers the actual enqueue/execution to phase 5") — this is not a new deviation, it is phase 5 fulfilling a deviation phase 2 already flagged and the master plan already anticipated resolving here.
2. **Boundary 2 — webhook intake connects to real enqueue, gated by outcome.** `services/commands/shopify/enqueue_or_record_shopify_webhook.py` (phase 4, implemented) is updated to call `create_instant_task(ctx.session, TaskType.SHOPIFY_PROCESS_WEBHOOK, payload=asdict(ShopifyProcessWebhookPayload(webhook_intake_id=intake_id)))` immediately after the existing `create_shopify_integration_event(...)` call, guarded by `if intake_status == ShopifyWebhookIntakeStatusEnum.RECEIVED:`. `intake_id` is the exact local variable already in scope at that point (`insert_result.scalar_one_or_none()` from the `INSERT ... ON CONFLICT DO NOTHING ... RETURNING` statement — see "Phase 4 implementation dependencies to verify before approval") — there is no ORM `ShopifyWebhookIntake` instance to reference by attribute, only this string id. The call is explicitly **not** reached when: the delivery is a duplicate (dedupe hit via `ON CONFLICT DO NOTHING`, `intake_id is None`, early return), the topic is unsupported (`status=IGNORED`), the shop integration is `disabled`/`uninstalled` (`status=IGNORED`), the shop domain is unrecognized (early return before reaching this code at all), or the HMAC/webhook-id checks failed (function raises before reaching this code) — matching the explicit instruction that duplicate/ignored/unknown-shop/invalid-HMAC cases must never enqueue processing. Every one of these exclusions is a `return`/`raise` that happens *before* the insertion point in the function's actual control flow, not a condition this plan has to add itself.
3. **Task type set.** Four `TaskType` members are added: `SHOPIFY_PROCESS_WEBHOOK = "shopify_process_webhook"`, `SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP = "shopify_sync_webhooks_for_shop"`, `SHOPIFY_REMOVE_WEBHOOKS_FOR_SHOP = "shopify_remove_webhooks_for_shop"`, `SHOPIFY_RECONCILE_SHOP = "shopify_reconcile_shop"`. **`SHOPIFY_RECONCILE_SHOP` is included, not deferred**, resolving a potential master-plan gap cleanly rather than by omission: the master plan's phase 5 "Owns" line explicitly lists a "reconcile" handler, but phase 3's `sync_shopify_webhook_subscriptions_for_shop` already *is* full desired-vs-installed reconciliation (it computes both a creation-eligible and a removal-eligible set and reconciles both — it is not a one-way "install only" operation). Rather than either (a) silently omitting the task type the master plan names, or (b) inventing new reconciliation logic that would duplicate phase 3's, this plan registers `SHOPIFY_RECONCILE_SHOP` with a handler that delegates to the exact same `sync_shopify_webhook_subscriptions_for_shop` command as `SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP` (see Implementation plan step 6). This gives the master plan's named task type a real, registered, non-dead handler (satisfying `51_worker_runtime.md`'s "no implicit task discovery... unknown task type is a hard failure" by ensuring every declared type is actually routable and handled) while adding zero new business logic — a future phase (e.g. a periodic reconciliation scheduler) gains a semantically distinct enqueue point without this plan guessing at scheduling requirements it has no mandate to build. No master-plan deviation is requested for the task-type set.
4. **Payload dataclasses and field naming.** New file `domain/execution/payloads/shopify.py`:
   - `ShopifyProcessWebhookPayload(webhook_intake_id: str)`
   - `ShopifySyncWebhooksForShopPayload(shop_integration_id: str)`
   - `ShopifyRemoveWebhooksForShopPayload(shop_integration_id: str)`
   - `ShopifyReconcileShopPayload(shop_integration_id: str)`
   All `@dataclass(frozen=True)`, matching every existing payload in `domain/execution/payloads/` (e.g. `UploadPayload`, `ReminderPayload`). Field naming follows the **existing Shopify command convention** established in phases 2-3 (`SyncShopifyWebhookSubscriptionsForShopRequest.shop_integration_id`, `RemoveShopifyWebhooksForShopRequest.shop_integration_id`) rather than inventing a `*_client_id` variant — using the identical field name means each handler can pass the payload's field straight into the existing command's `incoming_data={"shop_integration_id": payload.shop_integration_id}` with zero renaming. `webhook_intake_id` (not `webhook_intake_client_id`) follows the precedent set by `UploadPayload.pending_upload_id` — this codebase already uses bare `_id` suffixes (not `_client_id`) for several client_id-valued fields when the referenced entity's identity is unambiguous from context.
5. **`SHOPIFY_PROCESS_WEBHOOK` handler behavior — minimal, no business processor.** The handler: loads the `ShopifyWebhookIntake` row by `webhook_intake_id`; if `status` is already `PROCESSED`, `PROCESSING`, `FAILED`, or `IGNORED`, logs and returns (idempotency guard, matching the re-fetch-and-check-state pattern used by `handle_finalize_pending_step_completion`); otherwise sets `status=PROCESSING`, `processing_started_at=now`, increments `attempts`, commits; then — because no product/order business processor exists yet in any approved plan — immediately sets `status=PROCESSED`, `processed_at=now`, and records a `ShopifyIntegrationEvent(event_type=WEBHOOK_PROCESSED, severity=INFO, metadata_json={"processing_mode": "no_business_processor_yet", "topic": ..., "webhook_id": ...})`. **Reason**: this validates the full execution pipeline end-to-end (enqueue -> route -> claim -> handler -> terminal state) using real infrastructure, without pretending product/order business logic exists. A later, not-yet-planned phase that adds real product/order processing replaces or extends this handler's body — it does not need to touch the task type, payload, queue mapping, or worker registration, per `16_background_jobs.md`'s "extending the system" pattern.
6. **`SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP` / `SHOPIFY_REMOVE_WEBHOOKS_FOR_SHOP` handlers — thin delegation, no new logic.** Each handler deserializes its payload, opens its own DB session via `models.database.get_db_session()` (the actual helper this codebase's handlers and `worker_base.py` import — not the generic contract doc's `task_db_session`), constructs a `ServiceContext(identity={}, incoming_data={"shop_integration_id": payload.shop_integration_id}, session=session)`, and calls the existing phase-3 command function **directly** (`await sync_shopify_webhook_subscriptions_for_shop(ctx)` / `await remove_shopify_webhooks_for_shop(ctx)`) — **not** through `run_service`. `run_service` is the HTTP-facing error boundary that catches `DomainError` and converts it to a `StatusOutcome`; calling it here would swallow `ShopifyGraphQLError`/`ValidationError` instead of letting them propagate to `worker_base`'s own `_schedule_retry_or_fail`, which is exactly the retry/backoff mechanism this task is supposed to use. `identity={}` is safe because phase 3's commands never read authorization data from `ctx.identity` — they load the `ShopifyShopIntegration` row directly by `shop_integration_id` and derive `workspace_id`/actor fallbacks (`ctx.user_id or integration.updated_by_id or integration.created_by_id`) from the loaded row, a pattern phase 3 already designed to tolerate an unauthenticated/system caller.
7. **`SHOPIFY_RECONCILE_SHOP` handler — alias, not new code.** Same body as the sync handler (item 6), calling `sync_shopify_webhook_subscriptions_for_shop`. No separate function is written; the worker's `HANDLER_MAP` simply points both `TaskType.SHOPIFY_SYNC_WEBHOOKS_FOR_SHOP` and `TaskType.SHOPIFY_RECONCILE_SHOP` at the same handler function, matching this codebase's own existing precedent of one handler function serving multiple `TaskType` keys in a `HANDLER_MAP` (e.g. `notification_worker.py`'s `HANDLER_MAP` maps `NOTIFICATION`, `DELAYED_NOTIFY_TO_CUSTOMER`, and `DELAYED_BATCH_NOTIFICATION` all to `handle_notification`).
8. **Dedicated queue, dedicated worker.** All four Shopify task types map to a single new queue, `queue:shopify`, in `QUEUE_MAP` (`services/infra/execution/task_router.py`) — not reusing the existing shared `queue:tasks` that several unrelated domains (email sync, location tracker, step completion) already share. A new `workers/shopify_worker.py` registers only these four Shopify task types in its `HANDLER_MAP`, following the exact `workers/notification_worker.py` shape (`async def main(): await init_db(); await run_worker("queue:shopify", HANDLER_MAP)`, `if __name__ == "__main__": asyncio.run(main())`). This directly satisfies the master plan's explicit phase-5 requirement ("dedicated queue/worker path so Shopify jobs stay isolated from unrelated task workloads") and its drift-prevention note ("do not mix Shopify jobs into unrelated workers").
9. **Migration required, scope-bounded.** Adding new `TaskType` members requires an Alembic migration, because `task_type_enum` is a native Postgres enum (`SAEnum(TaskType, name="task_type_enum", create_type=True)` on `ExecutionTask.task_type`) — confirmed by inspecting `models/tables/execution/execution_task.py` and by the existing, repeated pattern of one-line migrations for every prior `TaskType` addition (e.g. `aa10c8c7e001_add_email_inbox_sync_task_type.py`: `op.execute("ALTER TYPE task_type_enum ADD VALUE IF NOT EXISTS 'email_inbox_sync'")`). This plan adds one migration with four such `ALTER TYPE ... ADD VALUE IF NOT EXISTS` statements, following the exact existing idiom — no new migration pattern is invented. This is explicitly within phase 5's mandate per the master plan's own "Must not touch: ... schema except execution enum migrations if not done earlier" — i.e. execution enum migrations are the one schema change phase 5 is expected to make.

## Risks and mitigations

- Risk: This plan's Boundary 2 insertion point is written against phase 4's actual confirmed code structure, but a future refactor of `enqueue_or_record_shopify_webhook.py` (outside this plan's scope) could silently move the `RECEIVED` branch and break the "insert immediately after `create_shopify_integration_event`" assumption.
  Mitigation: Implementation plan step 10 names the exact guard condition (`intake_status == ShopifyWebhookIntakeStatusEnum.RECEIVED`) rather than a line-number/position assumption, and the updated boundary test (step 11) asserts source-order precisely, so any future refactor that breaks this ordering fails a test rather than silently mis-enqueuing.
- Risk: Boundary 2's enqueue call is accidentally placed so it also fires for `IGNORED` or duplicate outcomes, causing phase 5 to "process" webhooks phase 4 deliberately marked as not-for-processing.
  Mitigation: Acceptance criteria item 7 and the corresponding boundary test explicitly assert zero tasks are created for every non-`RECEIVED` outcome; Implementation plan step 10 states the constraint explicitly rather than leaving it to be inferred from code structure alone.
- Risk: Updating phase 4's `test_webhook_intake_boundary.py` in this phase is mistaken for "modifying an archived plan" and skipped, leaving a test that fails the moment Boundary 2 is connected.
  Mitigation: "Resolved decisions" item 11 and Implementation plan step 11 make this an explicit, required step — only the phase-4 *plan document* is immutable once archived; its *test code* is expected to evolve as later phases build on top of it, exactly as this plan does.
- Risk: The sync/remove handlers call `run_service` instead of the command function directly, silently swallowing retryable `ShopifyGraphQLError`s into a `StatusOutcome` that the handler then ignores, so `worker_base` never retries a transient Shopify failure.
  Mitigation: "Resolved decisions" item 6 and Acceptance criteria items 9-11 make this an explicit, tested requirement — a test asserts a raised `DomainError` from the mocked command actually propagates out of the handler.
- Risk: `SHOPIFY_RECONCILE_SHOP` drifts into a "real" distinct reconciliation implementation later without anyone remembering it was originally just an alias, leading to two divergent reconciliation code paths.
  Mitigation: "Resolved decisions" item 7 documents the alias explicitly and in the worker's own `HANDLER_MAP` (both keys visibly point at one function), making the relationship self-evident from the code, not just this plan document.
- Risk: A new `queue:shopify` worker process is added to the Makefile but never actually started in any real environment (mirroring the existing gap where `notification-worker`/`presence-worker`/`analytics-worker` are Makefile-only with no `Procfile`/`docker-compose` entry), so Shopify tasks pile up as `OPEN`/`PENDING` forever outside local dev.
  Mitigation: This is explicitly flagged as an existing, pre-dating gap in "Resolved decisions" item 10 and deferred to master phase 7 by name, not silently repeated without acknowledgment — phase 7's plan should be written with this specific gap in mind, covering all of `notification`/`presence`/`analytics`/`shopify` workers' production wiring together, not just Shopify's.
- Risk: Adding four `ALTER TYPE ... ADD VALUE` statements in one migration hits a Postgres transactional-DDL edge case (e.g. `ADD VALUE` cannot be used in the same transaction as a later `commit()`-requiring statement in older Postgres versions).
  Mitigation: This exact pattern (one or more `ALTER TYPE ... ADD VALUE IF NOT EXISTS` statements, no other DDL in the same migration) is already used successfully by multiple prior migrations in this repository; this plan follows the identical, already-proven idiom rather than inventing a new migration shape.
