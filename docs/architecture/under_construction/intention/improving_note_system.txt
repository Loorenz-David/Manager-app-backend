
INTENTION:

We will now improve the current nove model TaskNote and routers for creating and updateing a note (route_create_note , route_update_note ).

The note model will have one more column called "users_read_list", that column will be a jsonb object holding a list of strings.
The note model will have one more column called "plain_text" that column will be a string

the note router and the service used for creating a note route_create_note support the incoming key "users_read_list"

The note "content" will be a list of objects because this content received by the create and update router will be a list of objects, like the case converstation router receives ( send_message_route ).
The note services for creating a note and updating it should have similar architecture as the case service send_message with a helper like write_case_message where the content is validated with the enum InputContentTypeEnum

We will create a router and service specialize on adding to the TaskNote.users_read_list list, the service use for this task router should be indepondent on the list of strings it stores. this service can be created at task service folder

We will create a router and service specize on getting the notes given a task id, this service should obtain all the notes and the image_links for each note, we use the note ids for obtaining those images through the image link model.
This service will have a serializer which serializes:
[{
note: client_id, task_id, note_type, content, created_at, created_by ( ### ), updated_at, updated_by ( ### ), plain_text, user_read_list 
note_images: image objects using the serializer serialize_image
}]
the note serializer can live at the task serializers, the current note serializer serialize_note will be replaced by that new serializer

the current get_task service will no longer bring the serialize notes 

the image link link model ( ImageLink ) will now support entity type of NOTE="note" in the ImageLinkEntityTypeEnum ( migration file should reflect this update ).

the image link model will now support a major_entity_type and a major_entity_client_id column like the NotificationPin model
